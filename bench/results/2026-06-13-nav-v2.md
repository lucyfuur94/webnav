# Navigation benchmark v2 (saucedemo) — webnav-walk vs raw-browser — 2026-06-13

Run of `docs/superpowers/specs/2026-06-13-navigation-benchmark-v2-design.md`.
Live browser + network (playwright-cli on PATH, saucedemo reachable). **Both arms on Haiku**
(`claude-haiku-4-5-20251001`) — the deliberate dogfood (cheap agent + deterministic walk vs cheap
agent ad-hoc-driving). 2 multi-step tasks × 2 arms × 3 trials + a no-op floor + anonymized per-answer
judge (25 agents total).

- **Arm A — webnav walk:** told to use `webnav walk`/`walk-resume` as the primary instrument.
- **Arm C — raw browser:** `use navigate/snapshot/click/type` only, no map. Same browser underneath.

## Headline (tool-calls = the clean cost signal)

| Task | Hops | Arm A (walk) median calls | Arm C (raw) median calls | A reached | C reached |
|---|---|---|---|---|---|
| **S2 — product detail** | 2 | **6** (5–10) | **18** (14–28) | 3/3 | 2/3 |
| **S3 — checkout overview** | 4 | 45 (28–52) | 42 (6–127) | 3/3 | 3/3 |

- **S2 is a clean win: webnav-walk used ~3× fewer agent steps (6 vs 18) and was more reliable
  (3/3 vs 2/3).** The walk drove login→inventory→detail; the raw agent ground through
  snapshot→read→click loops and one run failed on browser-session churn. This is the thesis working:
  fewer agent steps = fewer snapshots ingested = fewer reasoning tokens, on a genuine multi-hop route.
- **S3 did NOT separate (45 vs 42, both arms swung wildly 28–52 / 6–127)** — and the reason is the
  v2 design's predicted failure mode, observed live: **several arm-A runs ABANDONED the walk and drove
  manually** (`protocol-abandoned`), collapsing A into C; conversely the one 6-call arm-C run actually
  *picked up* `walk`. When the walk is used it's cheap (6 calls); when the agent falls back to manual
  driving the arms are identical. The separator is **whether the walk is actually used**, not capability.

## Honest caveats (do not over-read this)

1. **Judge verdicts are noisy on this rubric — ignore them as the cost signal.** The judge marked
   several bare-but-correct answers "wrong"/"partial" for "no live-evidence in the answer text," even
   when the agent's trace shows it genuinely navigated there (e.g. an A run that reached checkout-
   overview and reported the exact $43.18 was scored "wrong" for not narrating its evidence). That's a
   judge-formatting artifact, not a navigation failure. **Tool-calls + reached-goal are the trustworthy
   signals here; the judge needs a "trace-aware" prompt next run.**
2. **N=3 is small** and Haiku's strategy choice (walk vs manual) is itself stochastic — the S3 mud is
   partly that. The v2 design calls for N≥5 AND *forcing* the instrument harder; this run's arm-A prompt
   said "use the walk" but didn't hard-block the manual fallback, so the collapse still happened.
3. **The real, repeatable finding (matches the 2026-06-13 v1 run):** the walk wins decisively *when
   used* (S2; and the single 6-call S3 run), and the bottleneck is **walk ergonomics/discoverability** —
   agents default to the familiar manual loop. That's a product fix (make `walk` the obvious first move,
   add the bare-continue resume R5.1), not a capability gap.

## Verdict
On a clean 2-hop task, **webnav-walk cut agent steps ~3× and was more reliable** — the cost thesis
holds where the route is the work. On the 4-hop task the signal was swamped by agents abandoning the
walk for manual driving; that's a known ergonomics issue, not a webnav-can't. Next run: trace-aware
judge, N≥5, and a harder instrument-forcing prompt (or the R5.1 bare-continue resume) to stop the
arm-A→manual collapse. A genuinely honest result: **one clear win, one inconclusive — reported as such.**

# automationexercise.com — added 2026-06-13 (honest negative/inconclusive)

Same A-vs-C v2 harness, Haiku, 2 multi-step tasks × 3 trials + anonymized judge, with a HARDER
instrument-forcing arm-A prompt ("walk first, only fall back if it genuinely fails").

| Task | Arm A (walk) median calls | Arm C (raw) median calls | A reached | C reached | A actually used walk? |
|---|---|---|---|---|---|
| AE-detail (home→product detail) | 8 (8–15) | 10 (8–12) | 3/3 | 3/3 | **0/3** |
| AE-cart (home→cart) | 12 (4–12) | 4 (4–8) | 3/3 | 3/3 | **0/3** |

## The honest read: this run did NOT test the walk
**All 6 arm-A runs abandoned `walk` (`usedWalk` 0/6)** — even with the hardened prompt. The agents'
own notes repeatedly say *"webnav walk failed due to playwright-cli daemon issues, navigated directly
via playwright-cli."* So arm A collapsed into arm C, and the near-tie (8 vs 10) / loss (12 vs 4) is
**A≈C with different branding**, not a walk-vs-raw measurement. Two real causes, both worth fixing:
1. **`walk` was unreliable on this site under concurrent load** — daemon instability when 4+ live
   browsers run at once (the benchmark's own concurrency). The walk genuinely failed, so the agent
   *correctly* fell back (the prompt allowed it on real failure). That's a robustness gap, not the
   agent being lazy this time.
2. **The AE map's edge connectivity is thin** — the cross-link mesh only connected the always-present
   nav links; `product-details` is reachable but the route is short, so even a working walk saves
   little on these 1–2-hop tasks. AE-cart is a single nav-link hop — no regime for recall to win.

## Verdict (negative, reported as such)
**No webnav win on automationexercise.** The tasks were too shallow (1–2 hops) AND the walk didn't
run (daemon flakiness under load → fallback). This is the honest other side of the saucedemo S2 win:
webnav helps on DEEP, reliable multi-hop routes; on shallow routes, or when `walk` itself is flaky
under load, an agent ad-hoc-driving does just as well. Combined finding across both sites: **the
walk wins when (a) the route is genuinely deep AND (b) the walk is actually used and stable.** The
gating issues are walk reliability-under-load + ergonomics (agents fall back), not the core idea.
Next: stabilize the daemon under concurrency, deepen the benchmark routes, force/ease the walk (R5.1).
