# R1.1 — Live-data benchmark + 3rd arm (raw browser) — design

**Date:** 2026-06-02 · **Status:** approved (brainstorm), pending spec review
**Supersedes the task design of:** `docs/superpowers/specs/2026-06-02-r1-ab-benchmark-design.md`

## Why (the flaw R1 exposed)

R1's first live run (`bench/results/2026-06-02.md`) surfaced two methodology
flaws:
1. **Tasks were answerable from memory.** Several baseline subagents answered
   well-known facts (npm semver, HTTP 429, CAP theorem) with **0 tool calls** —
   so the token comparison was partly "model recall vs webnav navigation," not a
   clean tool-vs-tool comparison.
2. **Only two arms.** The most important comparison for webnav's thesis —
   *webnav's navigation skeleton vs an agent ad-hoc-driving the same browser* —
   was never run.

R1.1 fixes both: tasks that **force a live fetch**, and a **third arm** that gives
the agent the raw browser with no webnav map.

## Decisions (settled in brainstorm)

- **Every task must require a tool call.** Tasks ask for FRESH, SPECIFIC,
  VERIFIABLE data no model can recall (current GitHub numbers, latest
  release/commit, a live open-web value). If a task can be answered from memory,
  it is reworked or cut. Mixed flavors (github-live + web-live + botwalled).
- **Three arms**, all Sonnet:
  - **A — webnav:** the `webnav` CLI only (recall/search/route + evidence bundle).
  - **B — baseline:** Claude's WebSearch + WebFetch only.
  - **C — raw browser:** `playwright-cli` only (the same browser webnav uses
    underneath, but NO skeleton/recall/evidence). The agent navigates itself.
  **A vs C is the headline comparison** — it isolates what webnav's map adds over
  ad-hoc browser driving (the core CLAUDE.md thesis).
- **Rubric scoring (plausible + sourced), staleness-proof.** Live values change,
  so `gold_answer` is a RUBRIC, not a frozen value: *correct* = a specific current
  value with evidence it was fetched live (cites page/number/release/date),
  plausible, not a hedge; *partial* = right ballpark but unsourced / mild hedge;
  *wrong* = vague, stale/guessed, fabricated, or refused when the source was
  fetchable. This also punishes answering-from-memory (stale/unsourced → wrong).
- **Replace bench/ in place (R1.1).** Rewrite `bench/tasks.yml`, add Arm C to
  `bench/README.md`, keep the old `bench/results/2026-06-02.md` as a historical
  record of the flaw that motivated this. One canonical task set going forward.
- **Anonymized 3-way judging.** Judge sees three answers as "Answer 1/2/3", arm
  identity hidden, order varied per task; de-anonymized after.
- Judge is a BENCHMARK tool only — webnav stays zero-LLM (principle #5a). webnav
  engine is unchanged; R1.1 only touches `bench/`.

## Task set (`bench/tasks.yml`, rewritten — ~11 tasks)

Categories: `github-live`, `web-live`, `botwalled`.

**github-live** (current declared signals — webnav's strength):
- `gh-requests-issues` — open-issue count of `psf/requests` right now.
- `gh-httpx-release` — latest release tag/version of `encode/httpx` right now.
- `gh-stars-compare` — which has more stars right now, `pydantic/pydantic` vs
  `python-jsonschema/jsonschema`; give both current counts.
- `gh-tenacity-commit` — subject of the most recent commit on `jd/tenacity`'s
  default branch.
- `gh-fastapi-license` — license `tiangolo/fastapi` declares + rough current stars.

**web-live** (open-web; forces a fetch):
- `web-node-version` — current latest LTS Node.js version per nodejs.org.
- `web-python-version` — latest stable Python 3 release on python.org right now.
- `web-tsc-latest` — latest published version of the `typescript` npm package on
  npmjs.com right now.

**botwalled** (honest boundary retained):
- `bw-hn-top` — title of the current #1 Hacker News front-page story
  (news.ycombinator.com is automation-friendly — a fair live target for all arms).
- `bw-google-serp` — current top organic Google result for "best note taking app"
  (Google bot-walls — expect honest punts; catches fabrication).

(One github-live or web-live task may be dropped/added during authoring to land
near 11; the loader does not hard-code the count.)

## Arm prompts (committed VERBATIM in `bench/README.md`)

Arm A and Arm B prompts are unchanged from R1 (CLI-only / WebSearch+WebFetch-only).
**Arm C prompt (new):**

```
You are answering using ONLY the `playwright-cli` browser tool. You do NOT have
webnav, WebSearch, or WebFetch.

playwright-cli drives a real browser. Common commands (run via your shell):
  playwright-cli open <url>      # open the browser at a URL
  playwright-cli goto <url>      # navigate
  playwright-cli snapshot        # capture the page (text + element refs)
  playwright-cli click <ref>     # click an element
  (run `playwright-cli --help` for the full verb list)

Navigate to the relevant site yourself and read the answer off the live page.
Do NOT answer from prior knowledge — fetch it.

Task: <PROMPT>

Answer concisely. If you cannot retrieve it, say so honestly. End with:
ANSWER: <final answer in 1-3 sentences>
```

All three arms get the SAME `<PROMPT>` (fairness). Arm A and Arm C both drive
playwright-cli but in DISTINCT sessions (`-s=` names) so they never collide.

## Data flow (per task — 3 arms)

```
dispatch A (webnav), B (baseline), C (raw browser) concurrently, all sonnet
  → {answer, tokens} ×3
judge(gold_rubric, anonymized[A,B,C] randomized order, sonnet) → 3 verdicts
record row → aggregate → bench/results/<date>.md
```

## Report (`bench/results/<date>.md`)

- Headline: 3-way quality tally (A/B/C wins + ties) + median tokens per arm.
- **Key cut: A vs C** — "does webnav's skeleton beat raw browser driving?" on
  quality and tokens, especially on github-live.
- Secondary: A vs B (the original).
- Per-category table + per-task table (3 verdicts, 3 token counts) + caveats.
- Old `2026-06-02.md` retained unchanged as a historical record.

## Code impact (small)

- **`bench/load.ts`** — only change: `ALLOWED_CATEGORIES` →
  `['github-live', 'web-live', 'botwalled']`. Loader logic unchanged.
- **`tests/bench/load.test.ts`** — update fixtures to the new categories; the
  "rejects unknown category" test still asserts rejection.
- **`bench/tasks.yml`** — rewritten (content).
- **`bench/README.md`** — add Arm C prompt; recipe + report template go 3-arm.
- **Orchestration (live run)** — done by the main agent: ~11 tasks × 3 arms + 11
  judges, smoke-first then full.
- No `src/` changes.

## Testing

- TDD the `ALLOWED_CATEGORIES` change + its test (loader logic untouched).
- Validate the real rewritten `tasks.yml` loads (all new categories, ~11 tasks).
- The experiment is validated by RUNNING it (live), not by asserting agent
  output: smoke 1–2 tasks across all 3 arms, confirm a sensible 3-way scored row,
  then full run.

## Caveats (baked into the report)

- `playwright-cli` is on PATH; a raw-browser arm may hit bot-walls / render races
  / slow loads — those are REAL outcomes (Arm C losing there is legitimate signal,
  not a harness bug).
- Live values drift — rubric scoring handles it; report timestamps the run.
- One sample; judge fallibility; sample size ~11 — directional, not significance.

## Success criteria

1. A rewritten, re-runnable 3-arm benchmark whose every task forces a live fetch.
2. At least one full results report comparing A/B/C, with the A-vs-C cut explicit.
3. Honest finding on the real question: does webnav's navigation skeleton beat an
   agent ad-hoc-driving the same browser, on live data — on quality and/or tokens?
